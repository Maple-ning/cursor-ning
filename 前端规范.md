采用基于功能模块的目录结构，确保项目清晰、可维护。

目录职责说明
目录	职责	关键约束
api	所有后端接口定义，按模块划分，统一管理请求路径和参数类型。	禁止在组件内直接调用 axios/fetch，必须通过 api 层。
assets	存放编译时引用的静态资源（图片、字体、全局样式）。	图片大于 10KB 建议放入 assets，小图标优先使用 SVG 组件或字体图标。
components	可复用的 UI 组件，与业务逻辑解耦。	组件必须遵循单一职责，每个组件一个文件夹，包含样式、类型和测试文件（可选）。
config	项目配置，如路由表、环境变量映射、第三方服务配置。	禁止硬编码业务相关配置，必须统一管理。
hooks	封装可复用的逻辑（状态逻辑、副作用）。	以 use 开头，返回非 JSX 元素。
pages	路由对应的页面组件，按业务模块组织。	页面内可包含子组件，但仅限于该页面使用的私有组件。
stores	全局状态管理，按模块划分。	状态变更必须通过 actions/mutations，禁止直接修改 state。
utils	纯函数工具集（无副作用）。	必须包含单元测试，禁止引用 window 或 document 等特定环境 API（除非显式注明）。
types	全局类型声明（.d.ts）和公共 TS 接口。	API 响应类型统一以 ApiResponse<T> 包裹。
constants	枚举、常量配置（如权限码、状态值）。	禁止魔法值，所有枚举必须定义在 constants 中。
二、ESLint 配置规范
基于 @ali/eslint-config 或主流 airbnb/antfu 风格，结合阿里内部最佳实践。

2.1 安装依赖
bash
npm install -D eslint @typescript-eslint/parser @typescript-eslint/eslint-plugin eslint-plugin-react eslint-plugin-react-hooks eslint-plugin-import eslint-plugin-jsx-a11y
2.2 ESLint 配置文件（.eslintrc.js）

2.3 关键规则说明
规则	级别	说明
import/order	warn	强制导入顺序：外部库 → 内部模块 → 类型导入，提升可读性
react/function-component-definition	warn	统一使用箭头函数定义组件
@typescript-eslint/no-explicit-any	warn	避免滥用 any，需显式声明类型
no-console	warn/prod	开发环境允许，生产环境警告
import/no-cycle	error	禁止循环依赖，保证架构清晰

三、命名规范
3.1 文件命名
组件文件：PascalCase（如 UserCard.tsx）

页面文件夹：kebab-case（如 user-profile/）

工具/常量文件：camelCase（如 formatDate.ts）

类型文件：camelCase 或以 .types.ts 结尾（如 api.types.ts）

3.2 变量/函数命名
变量/函数：camelCase

常量：UPPER_SNAKE_CASE

枚举：PascalCase，枚举值 UPPER_SNAKE_CASE

接口：以 I 开头或直接 PascalCase（团队统一即可）
