## 1. 设计原则

1. **高内聚低耦合**：按业务模块组织代码，避免跨模块随意依赖。
2. **单一职责**：组件、函数、文件都应只做一件事。
3. **可测试性优先**：逻辑尽量函数化、纯净化，避免强耦合环境。
4. **一致性优先于个性**：统一命名、注释、提交规范。
5. **约束自动化**：能用 lint、format、CI 卡住的问题，不靠人工 review 兜底。

## 2. 目录结构规范

### 2.1 推荐结构（Feature First）

```text
src/
├── api/                      # 请求层（只处理 HTTP 与 DTO）
│   ├── modules/              # 按业务模块拆分接口
│   │   ├── user.ts
│   │   └── order.ts
│   ├── request.ts            # axios/fetch 实例、拦截器、统一错误处理
│   └── types.ts              # 接口请求/响应 DTO
├── assets/
│   ├── images/
│   ├── fonts/
│   └── styles/
│       ├── main.css
│       └── variables.css
├── components/               # 可复用公共组件
│   └── UserCard/
│       ├── index.vue         # React 项目可为 index.tsx
│       ├── types.ts
│       └── style.module.scss
├── composables/              # Vue 组合式逻辑（React 对应 hooks）
│   └── useAuth.ts
├── config/
│   ├── index.ts              # 应用配置（env 映射、超时、标题等）
│   └── routes.ts             # 路由表（可选）
├── constants/
│   ├── enum.ts
│   └── storageKeys.ts
├── layouts/
├── pages/                    # 路由页面，目录使用 kebab-case
│   ├── user-list/
│   └── order-detail/
├── services/                 # 业务层（组合多个 api + 业务规则）
│   └── userService.ts
├── stores/
│   ├── modules/
│   └── index.ts
├── types/
│   ├── api.ts
│   └── global.d.ts
├── utils/
│   ├── formatDate.ts
│   ├── storage.ts
│   └── validator.ts
├── App.vue                   # React 项目为 App.tsx
└── main.ts
```

### 2.2 目录职责与约束

| 目录 | 职责 | 关键约束 |
|---|---|---|
| `api` | 所有后端接口定义 | 禁止在页面/组件中直接 `axios/fetch` |
| `services` | 业务流程编排与数据转换 | 可以依赖 `api`，不反向依赖页面 |
| `pages` | 路由页面组件 | 页面私有组件可放同目录 `components` |
| `components` | 全局可复用组件 | 与业务解耦，推荐包含 `types` 与测试 |
| `stores` | 全局状态管理 | 状态变更通过 action/mutation，不直接改原始状态 |
| `utils` | 通用工具函数 | 尽量保持纯函数，避免副作用 |
| `constants` | 常量与枚举 | 禁止魔法值散落在业务代码中 |

## 3. 编码规范

## 3.1 命名规范

- 文件夹：`kebab-case`（如 `user-profile`）
- 组件名：`PascalCase`（如 `UserProfileCard`）
- 变量 / 函数：`camelCase`
- 常量：`UPPER_SNAKE_CASE`
- 枚举：枚举名 `PascalCase`，枚举值 `UPPER_SNAKE_CASE`
- 类型：`PascalCase`，文件建议 `*.types.ts` 或语义化命名

## 3.2 函数与模块规范

- 函数尽量短小，超过 80 行考虑拆分。
- 单文件导出数量建议可控，避免“工具杂货铺”。
- 复杂分支优先使用“提前返回”减少嵌套。
- 注释说明“为什么”，避免注释“做了什么”。

## 3.3 导入顺序规范

1. Node 内置模块
2. 第三方依赖
3. 项目内部模块（如 `@/`）
4. 同级与父级模块
5. 类型导入（`import type`）

---

## 4. 样式规范

- 推荐方案：`TailwindCSS` 或 `CSS Modules`，二选一为主、可混合。
- 页面样式就近维护，公共样式放 `assets/styles`。
- 避免全局污染：禁止无前缀全局类名泛滥。
- 主题变量（颜色、间距、圆角）统一放 design tokens（CSS 变量或主题配置）。
- 响应式断点统一管理，禁止“魔法断点”。

## 5. ESLint + Prettier 规范

### 5.1 目标

- ESLint 负责**代码质量与风险**。
- Prettier 负责**格式统一**。
- CI 必须执行 lint 与 type-check，未通过禁止合并。

### 5.2 推荐依赖

```bash
npm i -D eslint prettier eslint-config-prettier @typescript-eslint/parser @typescript-eslint/eslint-plugin
```

Vue 项目补充：

```bash
npm i -D eslint-plugin-vue @vue/eslint-config-typescript
```

React 项目补充：

```bash
npm i -D eslint-plugin-react eslint-plugin-react-hooks eslint-plugin-jsx-a11y
```

### 5.3 建议规则（核心）

| 规则 | 级别 | 说明 |
|---|---|---|
| `@typescript-eslint/no-unused-vars` | `error` | 禁止未使用变量（允许 `_` 前缀参数） |
| `no-console` | `warn` | 生产代码避免调试日志 |
| `no-debugger` | `error` | 禁止提交调试断点 |
| `import/no-cycle` 或同类规则 | `error` | 禁止循环依赖 |
| `import/order` | `warn` | 强制导入分组与排序 |
| `@typescript-eslint/no-explicit-any` | `warn` | 限制滥用 `any` |

### 5.4 `.prettierrc` 建议

```json
{
  "semi": true,
  "singleQuote": true,
  "tabWidth": 2,
  "trailingComma": "es5",
  "printWidth": 100
}
```

## 6. TypeScript 规范

- 禁止裸 `any`，优先 `unknown` + 类型收窄。
- API 响应统一包装：`ApiResponse<T>`。
- 公共 DTO 放 `api/types.ts` 或 `types/api.ts`。
- 开启严格模式（建议）：
  - `strict: true`
  - `noImplicitAny: true`
  - `noUncheckedIndexedAccess: true`

示例：

```ts
export interface ApiResponse<T> {
  code: number
  message: string
  data: T
}
```

## 7. API 与数据流规范

1. 页面仅调用 `services` 或 `store`，不直接请求接口。
2. `api` 负责请求与错误标准化，`services` 负责业务转换。
3. 统一处理鉴权、超时、重试、错误提示。
4. 写操作（POST/PUT/PATCH/DELETE）建议统一成功/失败反馈。

---

## 8. 状态管理规范

- 全局状态仅存放跨页面共享数据。
- 页面局部状态优先放组件内部。
- 异步逻辑统一放 action（Pinia）或 thunk/saga（Redux）。
- 禁止在多个 store 中重复维护同一源数据。

---

## 9. 测试规范

最低要求：

- `utils` 与 `services`：单元测试覆盖关键分支。
- 核心页面流程：至少 1 条集成/端到端测试。
- 新增 bug 必须附回归测试。

推荐工具：

- Unit: `vitest` / `jest`
- E2E: `playwright` / `cypress`

## 10. Git 与发布流程规范

### 10.1 分支模型

- `main`：稳定可发布
- `feat/*`：功能开发
- `fix/*`：缺陷修复
- `chore/*`：工程/依赖维护

### 10.2 Commit 规范（Conventional Commits）

```text
feat: 新增用户筛选功能
fix: 修复登录失效后重定向异常
refactor: 拆分文章列表数据转换逻辑
docs: 更新前端规范文档
```

推荐类型：

- `feat` `fix` `docs` `style` `refactor` `test` `chore` `perf` `build` `ci`

### 10.3 提交检查（推荐）

- `husky` + `lint-staged`：提交前执行格式化和 lint。
- `commitlint`：校验提交信息格式。
- CI：执行 `lint`、`type-check`、`test`、`build`。

## 11. 安全与性能基线

### 11.1 安全基线

- 禁止提交敏感信息（`.env`、私钥、token）。
- 前端只存必要信息，敏感信息尽量使用 HttpOnly Cookie 方案。
- 对用户输入进行校验与转义，防止 XSS。
- 第三方依赖定期升级并审计（`npm audit`）。

### 11.2 性能基线

- 路由级懒加载。
- 大列表虚拟滚动。
- 图片压缩、懒加载，优先现代格式（WebP/AVIF）。
- 避免重复请求，合理缓存与防抖节流。
- 定期使用 Lighthouse/性能面板做回归检测。
